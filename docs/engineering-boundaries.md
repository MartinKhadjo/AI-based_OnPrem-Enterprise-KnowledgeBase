# Engineering Boundaries

This repository is built for public professional visibility while protecting the actual software asset.

## Publicly safe information

The following information is safe and useful for recruiters and engineering reviewers:

- system purpose;
- high-level architecture;
- service roles and runtime boundaries;
- user workflows;
- operational principles;
- diagrams showing component responsibilities;
- non-sensitive technology categories;
- scalability and maintenance reasoning.

## Information kept private

The following items should remain outside the public repository:

- source code and commit history;
- exact prompts or prompt chains;
- API keys and `.env` files;
- model binaries or weights;
- raw user documents;
- processed databases, chunks and vector indexes;
- installer payloads;
- internal hostnames, IPs and deployment paths;
- internal logs and chat exports.

## Recommended professional disclosure path

For employers or technical interviewers, use a tiered disclosure model:

1. **Public repository:** architecture and portfolio evidence.
2. **Interview screen share:** deeper architecture walkthrough without transferring code.
3. **Private review:** selected snippets or code walkthrough only under NDA.
4. **Operational handover:** installer, documentation or source escrow only with written agreement.

This approach protects the intellectual property while still providing credible proof of engineering depth.
